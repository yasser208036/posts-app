## Story Goal

Add the friend-request domain so an authenticated user can discover other users, send a friend request, and accept/reject incoming requests, with friendships derived from accepted requests.

1. **List users** — `GET /api/users` returns every other user (never the caller), as `PublicUser`, so the UI can offer "send request" targets.
2. **Send request** — `POST /api/friends/requests` creates a `pending` request from the caller to a target user. Rejects self-requests (400), duplicate/pending requests (409), requests to a non-existent user (404), and requests where a friendship/reverse-pending already exists (409).
3. **List incoming requests** — `GET /api/friends/requests` returns the caller's `pending` **incoming** requests, each with the sender's public info.
4. **Accept / reject** — `POST /api/friends/requests/:id/accept` and `.../reject` transition a `pending` request the caller **received**; accepting creates the friendship (status `accepted`), rejecting sets `rejected`. Acting on someone else's request or a non-existent id returns 404.
5. **List friends** — `GET /api/friends` returns the caller's friends (the *other* party of every `accepted` request in either direction) as `PublicUser`.

**Not in scope:** presence/online status (Story 08), notification UI (Story 09), friends' posts and comments (Story 10), unfriending, blocking, cancelling a sent request, and any real-time transport (polling is added in later stories; this story is REST only).

## Backend Tasks

### 1 — Schema: `FriendRequest` model + `User` back-relations

**File: `backend/prisma/schema.prisma`** — add after the `Post` model (line 31), and add the two relation fields to `User` (inside the block at lines 10–18):

```prisma
model User {
  // …existing fields (id, name, email, passwordHash, provider, createdAt, posts)…
  sentRequests     FriendRequest[] @relation("SentRequests")
  receivedRequests FriendRequest[] @relation("ReceivedRequests")
}

model FriendRequest {
  id         String   @id @default(uuid())
  senderId   String
  receiverId String
  status     String   @default("pending") // "pending" | "accepted" | "rejected"
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt
  sender     User     @relation("SentRequests", fields: [senderId], references: [id], onDelete: Cascade)
  receiver   User     @relation("ReceivedRequests", fields: [receiverId], references: [id], onDelete: Cascade)

  @@unique([senderId, receiverId])
  @@index([receiverId, status])
  @@index([senderId, status])
}
```

Run `cd backend && pnpm prisma:migrate` (name it `add_friend_request`) then `pnpm prisma:generate`. The `@@unique([senderId, receiverId])` enforces "no duplicate requests" at the DB level (per acceptance criterion 2).

## Edge Cases & Failure Modes

- **Self-request** — `receiverId === req.userId` → 400 `{ message: "Cannot send a request to yourself" }`. Enforced in `friends.controller.sendRequest` before any DB write (acceptance criterion 2).
- **Duplicate request (sequential)** — `existingActiveRequest` returns true (pending or accepted, either direction) → 409. Enforced in `sendRequest`.
- **Duplicate request (concurrent)** — two simultaneous sends slip past the check; `@@unique([senderId, receiverId])` trips `P2002` → translated to 409 in the catch block (same pattern as `auth.controller.ts` lines 47–55).
- **Reverse pending** — B already sent A a pending request; A sending to B is blocked by `existingActiveRequest` (either-direction check) → 409. The UI (Story 09) should show B's request to A as acceptable rather than let A send a mirror request.
- **Accept/reject someone else's request** — `setRequestStatus` scopes `where { receiverId, status: "pending" }`; `count === 0` → 404 `{ message: "Request not found" }`. **Never 403** — existence is not leaked (matches the owner-scoping rule in `CLAUDE.md`).
- **Accept an already-handled request** — status no longer `pending` → `count === 0` → 404. Idempotent from the client's view.
- **Request to a non-existent user id** — `findUserById` returns null → 404 `{ message: "User not found" }`.
- **Deleted user** — `onDelete: Cascade` removes their `FriendRequest` rows automatically; `listFriends`/`listIncomingRequests` never dereference a missing party.
